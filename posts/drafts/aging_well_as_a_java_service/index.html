Aging well as a java service

Field Notes

Recently I got to work with this not very old (4 years codebase to be precise) java service and found it not to be in a great shape. My perspective (even though I’ve contributed some shitty code + not been a great steward of a codebase) has been that keeping a codebase readable (and structured well) is an obvious advantage because its better serves functionality since its a closer better to the actual domain and is the easier thing to do if you are owning the service in the medium to long term. Its also pretty evident when you use LLM agents that they just work much better on well structured codebases.
